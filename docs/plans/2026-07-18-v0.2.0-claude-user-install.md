# OnlyiFlow v0.2.0 — Claude User-Scope Installation Plan

Date: 2026-07-18

Owner direction: proceed with Claude Code user-scope persistent installation only. The previously
planned environment-free runtime and multi-Flow work are cancelled.

## Objective

Deliver one local Claude Marketplace package that installs OnlyiFlow at Claude Code `user` scope,
loads in every project on that Windows account while its local Marketplace directory and
user-selected Python environment remain available, remains explicitly invoked, and can be updated,
disabled, enabled, and uninstalled without changing any unrelated plugin or user configuration.

The existing source-Python runtime contract remains intentional:

```text
Claude Code
  -> installed OnlyiFlow plugin cache
  -> python from the host process environment
  -> plugin-owned server/stdio.py and src/onlyiflow
```

“Global installation” means user scope on one computer. It does not mean cross-computer sync,
system-wide installation for every Windows account, automatic invocation, or environment-free
execution.

## Decisions Replacing The Previous Plan

The following work is removed from v0.2.0 and is not automatically rescheduled:

- PyInstaller or another frozen runtime;
- a dedicated build environment or frozen-runtime dependency lock;
- Authenticode signing and Smart App Control acceptance;
- no-Python/no-Conda clean-machine acceptance;
- npm distribution or an installer;
- schema v2, workspace binding, multiple active flows, or owner CLI changes;
- macOS, Linux, Windows ARM64, or other runtime artifacts.

The cancelled frozen-runtime source changes, tests, generated executables, stop report, and
`onlyiflow-build` Conda environment were removed before this plan was written. The v0.1.0 release
tree was restored cleanly.

## Required User Environment

Every computer that installs the v0.2.0 Claude plugin must already have:

- Claude Code at a version that supports plugin Marketplaces and user-scope lifecycle commands;
- Python 3.11 or newer in an environment selected by the user;
- the selected environment's `python` command available to Claude Code processes;
- the dependencies listed in `requirements.txt`, currently `fastmcp>=3.4,<4`, installed in that
  selected environment.
- the extracted `onlyiflow-v0.2.0-claude-marketplace` directory retained at a stable local path
  after `claude plugin marketplace add` and plugin installation.

The plugin package contains OnlyiFlow's `server/` and `src/` trees. It does not require the
OnlyiFlow Git repository to remain available after installation, but Claude Code 2.1.197 was
observed resolving a directory Marketplace during session loading: hiding that Marketplace made
the installed Skill report `Unknown command`. Therefore v0.2.0 requires both the extracted local
Marketplace directory and the configured Python environment to remain available on every computer.

v0.2.0 documents this prerequisite and detects a missing environment during verification. It does
not select, create, repair, or update the user's environment and installs no dependency
automatically. Conda, virtualenv, system Python, and environment names remain user choices.

## Product And Security Invariants

The v0.1.0 workflow contract remains unchanged:

- exactly seven MCP tools;
- the same closed schemas, state transitions, privacy rules, and error mapping;
- explicit invocation only;
- ordinary coding requests produce zero OnlyiFlow activation;
- quick work in a managed project retains the two-call path;
- no Hooks, subagents, commands, background monitors, daemons, adapters, or generated Agent
  configuration;
- no model-callable owner approval;
- no claim that the plugin prevents direct Git operations.

The installation work must not alter any non-OnlyiFlow Claude plugin, Marketplace, cache, setting,
or process. All lifecycle paths use exact OnlyiFlow identifiers and run cleanup in `finally`-style
boundaries.

## Source And Package Layout

Persistent user installation makes repository auto-discovery observable. Claude Code can discover
root `skills/` and project `.mcp.json` in addition to an installed user plugin, which can expose the
same Skill or server twice when the OnlyiFlow repository itself is opened.

Move only host packaging resources below a non-auto-discovered template root:

```text
packaging/
  codex/
    .codex-plugin/plugin.json
    .mcp.json
    skills/onlyiflow/
  claude/
    .claude-plugin/plugin.json
    .mcp.claude.json
  zcode/
    .zcode-plugin/plugin.json
  shared/
    skills-claude/onlyiflow/
```

The repository root retains `server/`, `src/`, tests, scripts, and project metadata. It no longer
acts as an accidental Claude project plugin. The candidate builder reads the packaging templates
and copies the common source runtime into each generated host root.

Codex and ZCode generated candidates must remain behaviorally identical to v0.1.0 apart from the
declared version. No Codex or ZCode installation lifecycle is part of v0.2.0.

## Target Artifact

The generated roots become:

```text
build/loader-candidates/
  codex-marketplace/
    .agents/plugins/marketplace.json
    plugins/onlyiflow/
  claude-marketplace/
    .claude-plugin/marketplace.json
    plugins/onlyiflow/
  zcode/
    marketplace.json
    onlyiflow/
```

The Claude Marketplace uses the stable local name `onlyiflow-local` and contains exactly one
plugin entry:

```json
{
  "name": "onlyiflow-local",
  "owner": {"name": "OnlyiFlow"},
  "description": "Local user-scope distribution for OnlyiFlow.",
  "plugins": [
    {
      "name": "onlyiflow",
      "source": "./plugins/onlyiflow",
      "version": "0.2.0",
      "strict": true
    }
  ]
}
```

The plugin entry and `.claude-plugin/plugin.json` versions must agree. The installed plugin remains
self-contained at the source level and launches only through `${CLAUDE_PLUGIN_ROOT}`:

```text
plugins/onlyiflow/
  .claude-plugin/plugin.json
  .mcp.claude.json
  skills-claude/onlyiflow/SKILL.md
  pyproject.toml
  server/stdio.py
  src/onlyiflow/
```

The proposed release asset is `onlyiflow-v0.2.0-claude-marketplace.zip`. It is a local Marketplace
archive, not a public Marketplace submission or npm package.

## Execution Tasks

### v0.2-T1 — Freeze installation contracts with failing tests

Add tests that initially fail for the current v0.1.0 layout:

- the repository root contains no host Skill or MCP auto-discovery resources;
- the builder reads host templates from `packaging/`;
- the Claude output root is a Marketplace with exactly one relative-path plugin source;
- Marketplace and plugin versions agree;
- the installed candidate calls the user-selected `python`, retains `${CLAUDE_PLUGIN_ROOT}`, and
  includes `requirements.txt`;
- Codex and ZCode candidates retain their current launcher contracts;
- no generated candidate contains another host's manifest or Skill wrapper;
- rebuilding to a fresh output root is deterministic at the file-content level.

Verification: each test fails for its intended v0.1.0 reason before production files move.

### v0.2-T2 — Generate the Claude Marketplace without changing runtime behavior

Move host templates to `packaging/`, update `scripts/build_loader_candidates.py`, and generate the
new Claude Marketplace root. Keep candidate assembly offline and dependency-free.

Verification:

- focused packaging and launcher tests pass;
- `claude plugin validate` accepts both the Marketplace and its plugin;
- the generated Claude stdio server initializes and lists the exact seven tools;
- the complete existing test suite remains green;
- Codex and ZCode generated files still satisfy their v0.1.0 contracts.

### v0.2-T3 — Prove the user-scope lifecycle

Add one bounded lifecycle runner with testable command construction and unconditional OnlyiFlow
cleanup. Before mutation it snapshots all installed Claude plugins and Marketplaces and refuses to
proceed if an unmanaged OnlyiFlow installation is already present.

Run this exact sequence against a disposable copy of the local Marketplace:

1. add `onlyiflow-local` at user scope;
2. install `onlyiflow@onlyiflow-local` at user scope;
3. confirm it is enabled and its install path is under Claude's versioned plugin cache;
4. start a fresh unrelated project and perform a real MCP initialize, tools/list, and explicit
   quick flow;
5. directly verify that the cached package is self-contained at the runtime-file level while
   recording that Claude session loading still requires the directory Marketplace to remain;
6. disable the plugin and prove the Skill/MCP are unavailable;
7. enable it and prove they return;
8. exercise an actual version update through a disposable higher-version Marketplace copy and
   prove the active cache version changes;
9. uninstall the plugin and remove only the `onlyiflow-local` Marketplace;
10. confirm the Skill, MCP server, active cache entry, process, and temporary workspace are absent;
11. confirm every pre-existing non-OnlyiFlow plugin and Marketplace is unchanged.

Claude-owned inactive orphan cache behavior must be recorded separately from active installation
state. Precise deletion is allowed only for OnlyiFlow cache entries created by this runner.

### v0.2-T4 — Prove explicit behavior across projects

With the user plugin installed, use fresh Claude sessions in three unrelated temporary projects:

- an unmanaged empty project;
- a managed project with a quick flow;
- the OnlyiFlow source repository, which must expose exactly one installed Skill and one MCP server.

Acceptance:

- ordinary prompts do not load the OnlyiFlow Skill or call its MCP server;
- explicit `/onlyiflow:onlyiflow` prompts produce the expected real MCP call sequence;
- owner confirmation is still required before initialization;
- the user-installed plugin works in every project while the retained Marketplace is available;
- no prompt, response, or tool result exposes an external absolute path, credential, environment
  dump, command output, or transcript;
- network/model timeout is classified as infrastructure and stops further model traffic after the
  first confirmed occurrence.

The accepted v0.1.0 Skill quality and efficiency budgets remain regression ceilings. The cancelled
frozen-runtime size and cold-start budgets no longer apply.

### v0.2-T5 — Release-readiness evidence and handoff

After T1 through T4 pass:

- run the full unit suite;
- run the Codex Skill validator on the generated Codex candidate;
- run the Claude validator on the generated Marketplace and plugin;
- verify generated manifests contain version `0.2.0` consistently;
- verify the Git worktree contains no generated candidates, caches, databases, temporary projects,
  or lifecycle state;
- update `docs/product-spec.md`, `docs/engineering-spec.md`, `docs/release-guide.md`, and
  `AGENTS.md` only to describe the accepted Claude user-install behavior and retained environment

- write one accepted evaluation report with commands, hashes, lifecycle state, real MCP evidence,
  model evidence, and cleanup evidence;
- assemble and checksum `onlyiflow-v0.2.0-claude-marketplace.zip` without publishing it.

Only after every acceptance item passes may the owner be asked for separate authorization to
commit, push `main`, create tag `v0.2.0`, and publish a GitHub Release. No public plugin Marketplace
or npm publication is authorized.

## Stop Conditions

Stop immediately and preserve exact evidence if:

- Claude cannot load the user-installed Skill or MCP while the retained local Marketplace is
  available;
- user-scope installation duplicates the Skill or MCP server inside the source repository after
  template relocation;
- solving installation requires changing the seven-tool workflow semantics;
- cleanup would require deleting or modifying an unrelated plugin, Marketplace, cache, setting, or
  process;
- Claude's actual Marketplace or lifecycle contract differs materially from the documented and
  locally observed CLI behavior;
- the selected Python 3.11+ environment with `requirements.txt` installed cannot start the
  generated cached server;
- one confirmed network/model infrastructure timeout occurs during model-backed acceptance.

## Completion Definition

v0.2.0 is complete only when the Marketplace installs at Claude user scope, works in all three
project contexts while its local Marketplace directory is retained, remains explicit-only, passes
disable/enable/update/uninstall, leaves no active OnlyiFlow lifecycle state, preserves every
unrelated Claude component, passes the complete local verification suite, and has an accepted
evidence report.

Environment-free execution and multiple active flows are not implied by this completion claim.
