# OnlyiFlow 0.2.0 Claude User-Scope Installation Evidence

Date: 2026-07-18

Status: accepted release candidate committed to `main`; not tagged, released, or published

## Accepted Scope

Version 0.2.0 adds one local Claude Marketplace and a verified Claude Code `user`-scope lifecycle.
The workflow runtime, seven-tool MCP surface, one-active-flow rule, and explicit-only Skill behavior
remain unchanged from 0.1.0.

The accepted installation contract is deliberately bounded:

- installation applies to one Windows user account and works across that account's projects;
- the extracted local Marketplace directory must remain available while the plugin is installed;
- Python 3.11+ from a user-selected environment and the dependencies listed in `requirements.txt`
  remain prerequisites;
- environment-free execution, frozen binaries, signing, npm distribution, cross-computer setup,
  multiple active flows, and public Marketplace publication are not part of 0.2.0.

## Implementation Evidence

The repository root no longer contains host auto-discovery resources. Host templates live under
`packaging/`, and `scripts/build_loader_candidates.py` emits these isolated roots:

```text
build/loader-candidates/codex-marketplace/plugins/onlyiflow/
build/loader-candidates/claude-marketplace/plugins/onlyiflow/
build/loader-candidates/zcode/onlyiflow/
```

The Claude Marketplace is `onlyiflow-local`, contains exactly one relative-source plugin entry,
and agrees with the plugin manifest on version `0.2.0`. Its installed package contains exactly one
manual-only Skill, one MCP server, the bundled `server/` and `src/` runtime, and the generic Python
launcher contract.

The finalized launcher contract is `python -B <plugin-root>/server/stdio.py`. Every generated host
package includes `requirements.txt`, whose runtime dependency list matches `pyproject.toml`. The
host resolves `python` from its own process environment; no manifest contains `conda`, an
environment name, or an absolute interpreter path.

`scripts/run_claude_user_install_lifecycle.py` owns exact OnlyiFlow lifecycle identifiers, snapshots
unrelated Claude state, refuses an ambiguous baseline, and performs unconditional bounded cleanup.
`scripts/run_claude_user_install_acceptance.py` uses fresh non-persistent Claude sessions without a
side-load path and stops after the first non-passing model case.

Claude Code 2.1.197 declares `--allowedTools` as a variadic option. A first test run placed the
prompt after that option; the CLI consumed it and exited before network or MCP activity with
`Input must be provided`. The command builder now places the prompt before variadic options, and a
focused regression test freezes that order.

## Directory-Marketplace Limitation

A controlled negative run hid the disposable directory Marketplace after installation. The
installed inventory still reported one Skill and one MCP server, but a fresh Claude session
returned `Unknown command: /onlyiflow:onlyiflow` with zero MCP calls and zero project mutation.

This is accepted as a host installation prerequisite, not as an environment-free capability. The
final model acceptance therefore retained the directory Marketplace for every session. The
product, engineering, release, plan, README, and repository-agent documents all state this limit.

One model-acceptance attempt passed the unmanaged case and then encountered a fresh Claude session
that exposed only `WaitForMcpServers` and `Stop`, despite reporting the MCP layer ready. It made zero
OnlyiFlow calls and zero project mutations. This was classified as host tool-visibility
infrastructure, a regression test was added for the observed Chinese error wording, and the runner
stopped further traffic. One controlled rerun then passed all four cases.
