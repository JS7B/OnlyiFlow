# OnlyiFlow 0.3.0 Gate Configuration Evidence

Date: 2026-07-19

Status: released as v0.3.0 after local, Claude, and owner-assisted ZCode acceptance passed; Codex
live verification deferred by owner

## Accepted Scope

Version 0.3.0 adds an owner-confirmed Gate configuration step and a compact Gate-readiness view.
It expands the portable MCP surface from seven to eight deterministic tools while retaining the
existing one-active-flow rule, explicit-only activation boundary, and host-owned implementation
model.

The increment adds no dependency, storage-schema migration, background process, model call, or
network operation. Gate configuration is project-local and transport-independent.

## Runtime Contract

The eighth tool is `gate_configure`. It accepts a complete replacement list of one to 32 checks,
using the same bounded check identifiers, command argument counts, and timeouts as Gate execution.
It is available for a managed project with no active flow and, as a migration exception, for the
first non-empty Gate of an older active flow whose configuration is still empty.

The runtime now provides these guarantees:

- `project_status.data.gate_config` reports `configured`, `check_count`, and `required_count`;
- `project_init` directs the owner to Gate configuration before the first flow;
- `flow_start` returns `gate_checks_missing` until at least one check is configured;
- an older active flow with an empty Gate reports `gate_configure`, accepts one owner-confirmed
  initial configuration, and then resumes its state-specific action;
- a configured Gate remains locked against replacement while any flow is active;
- invalid input and active-flow conflicts preserve the existing configuration byte-for-byte;
- a valid replacement is written through a same-directory temporary file, flushed, synchronized,
  and atomically replaced;
- temporary files are removed after successful or failed replacement;
- responses expose check IDs, required flags, timeouts, and counts without returning commands;
- SQLite, domain events, Gate evidence, and model reports do not retain command text; and
- an already configured quick flow still reaches `implementing` through exactly
  `project_status -> flow_start`.

The Codex, Claude Code, and ZCode candidates contain the same bundled runtime and the same
owner-confirmation instructions. Their manifests all declare version 0.3.0.

## Test-First Evidence

Focused tests first froze the missing behavior, including readiness reporting, invalid-input
preservation, active-flow locking, successful atomic replacement, replacement-failure rollback,
command privacy, and the pre-flow configuration requirement. The implementation then made those
tests pass with the smallest shared runtime and MCP changes.

The documentation acceptance test also failed before this evidence file existed and passed after
the release guide and candidate record were completed.

The final source-flow check exposed an upgrade deadlock that the original greenfield tests did not
cover: a pre-0.3 active flow could have an empty Gate, causing status to direct `gate_run` while the
run returned `gate_checks_missing` and configuration remained locked. A focused regression test
first reproduced that failure. The transport-independent runtime now directs the legacy flow to
the normal owner-confirmed `gate_configure` boundary, permits only its first non-empty Gate, and
locks all later active-flow replacements.

A completion audit then found that the historical release-smoke runner still configured its Gate
through the controller immediately after initialization. A new failing test froze the required
two owner turns, and the runner was changed to model-visible proposal and confirmation sessions.
It no longer calls the fixture Gate configurator in that path.
